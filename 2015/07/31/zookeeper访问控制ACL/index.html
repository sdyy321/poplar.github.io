zookeeper访问控制ACL

文章目录
  1. 1. ACL
  2. 2. ACL机制
  3. 3. 代码实现

zookeeper提供了访问控制机制，用来实现多项目多集群共享同一个zookeeper后权限隔离，比如zookeeper下某个节点只允许集群A或项目B来操作，其他人只能查询不能做其他操作，可以有效避免一些误操作。

ACL

在zookeeper中，节点的ACL是没有继承关系的，是独立控制的。zookeeper的ACL通常表示为scheme:id:permissions，即分为三个维度，一个是scheme、一个是id、一个是permissions，下面我们会一一介绍

ACL机制

zookeeper的ACL表示为scheme:id:permissions

scheme字段：表示采用哪一种机制来进行访问控制

id字段：表示用户

permissions字段：表示相关权限(如只读、读写、管理等)

下面给出scheme字段可用的可选项：

类型 作用
world 表示所有人都可以访问，它下面的id用户只有一个即anyone
auth 表示需要通过用户验证，它不需要id，只要是通过验证的用户都有权限
digest 表示需要通过用户名密码方式的auth验证，id格式为username:BASE64(SHA1(password))
ip 表示需要通过IP校验，id为客户机的IP地址
super 表示超级用户，对应的id用户拥有超级权限，可以做任何事情

下面给出permissions支持的权限

权限 作用
CREATE 创建权限，可以在当前节点下创建子节点，即create(子节点)
DELETE 删除权限，可以删除当前的节点，即delete(子节点)
READ 读权限，可以获取当前节点的数据，可以遍历当前节点所有的子节点，即getChildren/getData
WRITE 写权限，可以想当前节点写数据，即setData
ADMIN 管理权限，可以设置当前node的权限，即setACL

代码实现

我们使用curator作为我们的zookeeper客户端，下面的代码都是基于curator做测试

  • world实现

    上例中，我们使用world机制来实现了一个ACL节点，通过ZooDefs.Ids的枚举你可以选择想要的ID，比如ZooDefs.Ids.ANYONE_ID_UNSAFE表示Id("world", "anyone")，ZooDefs.Ids.AUTH_IDS表示Id("auth", "")，ZooDefs.Ids.OPEN_ACL_UNSAFE表示ACL(Perms.ALL, ANYONE_ID_UNSAFE)，ZooDefs.Ids.CREATOR_ALL_ACL表示ACL(Perms.ALL, AUTH_IDS)，ZooDefs.Ids.READ_ACL_UNSAFE表示ACL(Perms.READ, ANYONE_ID_UNSAFE)

  • auth实现

    上例中我们通过auth机制，实现了某个节点只对当前通过auth认证的用户拥有对应的权限，其中最重要的代码是ZooDefs.Ids.AUTH_IDS,这个ID表示的就是当前通过验证的用户即admin:manager，当使用其他用户名密码认证建立连接后便无法操作该节点了，但是需要注意的是exists操作和getAcl操作并不受ACL许可控制，因此任何客户端可以查询节点的状态和节点的ACL。

    上例中，我们使用CuratorFrameworkFactorybuild了一个CuratorFramework，然后新建了个节点/test/test1/test2，这个节点的权限scheme:id:permissions设置为了digest:admin:manager:ALL，ACL机制为digest机制，只有通过用户名密码验证的用户才能查询操作该节点。

  • ip实现

    上例中我们设置了/mytest/test节点ACL为ip:127.0.0.1:cdrwa，只有本地机器可以访问该节点，服务端会校验客户端的真实IP地址，所以无论你如何设置authorization都不能手动修改你的ip地址，IP策略一般在多个集群公用同一个zookeeper服务时非常有效，这样可以在客户端不知情的情况下隔离了集群资源，缺点是可能你需要指定很多IP。

    在zookeeper最近几个版本中貌似去掉了host策略以及ip段策略，这里就不多介绍这2个策略了，有兴趣的可以研究下较早版本的zookeeper
